# Design Note

The part I took most seriously in this task was that PPP is asking for judgment support, not just text generation. The goal is not simply to summarise five profiles. It is to turn uneven public evidence into a usable call-priority tool: something structured enough for downstream review, careful enough not to overclaim, and commercially useful enough to help a consultant decide who deserves an early call, who belongs in calibration, and who should stay in mapping.

I chose a two-step Claude workflow for that reason. In the first step, Claude works in research mode with a real Anthropic tool call. The model is required to call `normalize_candidate_evidence`, Python executes the tool, and the tool result is fed back into the Claude conversation. That phase does not try to write the final brief; it only produces a normalized evidence object with a stable confidence state, recruiter-relevant signals, and explicit uncertainty notes. In the second step, Claude is called again without tools and must return exactly one PPP-compatible `CandidateBrief` JSON object. I found this more reliable than a single large prompt because enrichment, evidence calibration, and final formatting are different jobs, and because that separation makes the tool step materially improve the final shortlist note rather than sit beside it.

That architecture matters in executive search because a briefing is not a résumé summary. PPP needs a note that helps a consultant judge lane, seniority, likely relevance to the mandate, call priority, and what should be tested live. The trade-off I chose was to keep the flow simple and explicit rather than more agentic, because preserving those distinctions mattered more than adding complexity.

Strict schema alignment is also a business requirement, not just a technical preference. `output.json` is the actual deliverable, so I treated the export contract as non-negotiable: exactly five candidates, no extra fields, bounded scores, and numeric `current_role.tenure_years` for every record. When chronology is weak, the system keeps the number conservative and carries uncertainty in the prose instead of breaking schema. For this task, I preferred that trade-off over a more expressive but less dependable output shape.

The evidence model is deliberately conservative. Public profiles are incomplete, and identity confidence is not uniform across all five inputs. The pipeline therefore normalizes candidates into confidence-aware states and lets those states affect tone, score calibration, tenure handling, and outreach posture. A stronger match can be framed as a shortlist-style or adjacent-screen candidate. A weaker identity case can still be useful as a market-map lead, but I did not want it to read as equally verified.

I also tuned the output for recruiter use rather than polished prose. `role_fit.justification` and `outreach_hook` are meant to answer practical questions: why this person is worth a call now, how they compare with the rest of the slate, what the main fit is, what the key gap is, and what should be pressure-tested first. The system also keeps firm context qualitative when exact AUM is not defensible and avoids overclaiming chronology in low-confidence identity cases.

One trade-off worth stating directly is reproducibility. PPP wants recruiter-useful output based on public evidence, but public-web evidence is not perfectly stable and the final candidate briefs are still model-generated. For that reason, the repository supports both a fixture-backed review path and a live public-web path that demonstrates real external tool capability. I treated the fixture-backed path as the canonical evaluation path because it keeps the evidence pack, schema checks, and QA flow stable, while the committed `data/ppp/output.json` captures the strongest promoted artifact to assess.

That is a product and engineering choice, not a capability concession. The repository still supports live public-web enrichment and real Claude tool use, but submission evaluation should anchor to the best artifact that preserves recruiter usefulness, evidence discipline, and deliverable stability. More aggressive variants were only worth promoting if they kept those three qualities intact; otherwise they remained exploration-only.

With more time, I would improve three things. First, I would add a citation-aware reviewer view so a consultant could trace each sentence in the brief back to its supporting source or normalized claim. Second, I would make the QA layer more mandate-specific, especially around institutional versus wholesale depth, team leadership evidence, and product breadth. Third, I would add a consultant feedback loop so PPP could mark outputs as helpful, too cautious, or misleading and use that feedback to tighten calibration.

If I were building one additional automation for PPP, it would be a mandate-to-longlist triage workflow. Candidate briefing is useful, but the bigger operational gain is helping consultants decide who deserves immediate attention, who is adjacent but worth screening, and who should remain in the market map. Extending the current evidence normalization and role-fit logic into ranked outreach triage would fit naturally with PPP’s workflow, reduce first-pass research time, and make immediate-outreach versus map-only decisions more repeatable across mandates.
